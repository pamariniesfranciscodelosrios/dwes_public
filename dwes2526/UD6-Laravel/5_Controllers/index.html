1. Introduction (10 mins)

  • In Laravel, controllers are classes that group related request-handling logic into a single place.
  • They act as the link between the routes and the logic (image) of your application.
  • Laravel follows the MVC (Model-View-Controller) architectural pattern, which separates concerns to make code more maintainable.
  • Controllers are part of the "Controller" layer in this architecture.
  • Using controllers allows developers to write cleaner and more organized code compared to placing all logic inside route closures.

Step 4: Define the index method in UserController.php

namespace App\Http\Controllers;

use Illuminate\Http\Request;

class UserController extends Controller
{
    public function index()
    {
        $users = [
            ['name' => 'Alice', 'email' => 'alice@example.com'],
            ['name' => 'Bob', 'email' => 'bob@example.com'],
        ];

        return view('users.index', ['users' => $users]);
    }
}

In this method, we prepare some example data and pass it to the view using the view() helper.

3. Key Concepts

3.1. Returning a View Without a Controller

  • As we saw, it is possible to return a view directly from a route , without using a controller.
  • This is useful for simple pages like static landing pages, FAQs, or terms and conditions that donâ€™t require logic or data processing.

Hereâ€™s how you can do it:

With Route:view

Inside your routes/web.php file:

Route::view('/welcome', 'welcome');

This will render the resources/views/welcome.blade.php view when the user visits /welcome.

With Data

You can also pass data to the view like this:

Route::view('/about', 'about', ['name' => 'John Doe']);

Then in your Blade file (resources/views/about.blade.php), you can use the data:

<h1>Welcome, {{ $name }}</h1>

When to Use This

Use route-based views when:

  • The page is static or very simple.
  • Thereâ€™s no need for dynamic logic or complex data retrieval.
  • You want quick prototyping without creating a controller.

However, for most dynamic pages or anything involving logic, validation, or model interaction, you should use a controller.

Instead of defining all of your request handling logic as closures in your route files, you may wish to organize this behavior using "controller" classes. Controllers can group related request handling logic into a single class. For example, a UserController class might handle all incoming requests related to users, including showing, creating, updating, and deleting users. By default, controllers are stored in the app/Http/Controllers directory

What is a Controller?

A controller in Laravel is a PHP class that handles the incoming HTTP requests to your application. Instead of defining request logic in route files, it's better to move this logic to controllers, which makes your code cleaner and better organized. Each method in a controller corresponds to an "action" in response to a specific request.

Anatomy of a Controller

Controllers are typically stored in the app/Http/Controllers directory. Each controller class uses the Controller base class. Inside, you define public methods (actions) which Laravel will call when the associated route is matched. These methods can receive parameters, handle business logic, and return responses or views.

Types of Controllers

Laravel supports different controller types to suit different needs:

  1. Basic
  2. Resource
  3. Invokable

Basic Controller

Created using:

php artisan make:controller ExampleController

This is the default, where you define only the methods you need.

Resource Controller

Useful for CRUD operations. It automatically provides standard methods like index, create, store, show, edit, update, and destroy.

php artisan make:controller ProductController --resource

Laravel routes can be auto-generated for these using:

Route::resource('products', ProductController::class);

Invokable Controller

  • This controller has only one method: __invoke().
  • Ideal for single-action controllers.
php artisan make:controller ReportController --invokable

Used like:

Route::get('/report', ReportController::class);

Dependency Injection

Laravel supports automatic injection of dependencies into controller methods. For example, you can inject the Request object or a model directly:

public function store(Request $request)
{
    // You can access form data here
}

This helps in writing more testable and decoupled code.

Returning Different Responses

In a controller, you can return:

  • A Blade view: return view('home');
  • A redirect: return redirect('/dashboard');
  • A JSON response: return response()->json(['key' => 'value']);
  • A custom response with headers or status codes

Controllers allow full control over what is returned to the client.

3.2. Best Practices

To keep controllers maintainable, it's recommended to keep them focused and not overloaded with logic.

  • Use services or repositories to move complex business logic out of controllers.
  • Stick to naming conventions and RESTful practices, especially when building APIs.
  • For form validation, Laravel encourages the use of Form Request classes , which cleanly separate validation from your controller methods.
